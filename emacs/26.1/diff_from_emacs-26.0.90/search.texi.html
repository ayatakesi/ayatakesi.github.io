0005: @node Search
0006: @chapter Searching and Replacement
0007: @cindex searching
0008: @cindex finding strings within text
0009: 
0010:   Like other editors, Emacs has commands to search for occurrences of
0011: a string.  Emacs also has commands to replace occurrences of a string
0012: with a different string.  There are also commands that do the same
0013: thing, but search for patterns instead of fixed strings.
0014: 
0015:   You can also search multiple files under the control of @code{xref}
0016: (@pxref{Identifier Search}) or through the Dired @kbd{A} command
0017: (@pxref{Operating on Files}), or ask the @code{grep} program to do it
0018: (@pxref{Grep Searching}).
0019:

0022: * Nonincremental Search::     Specify entire string and then search.
0023: * Word Search::               Search for sequence of words.
0024: * Symbol Search::             Search for a source code symbol.
0025: * Regexp Search::             Search for match for a regexp.
0026: * Regexps::                   Syntax of regular expressions.
0027: * Regexp Backslash::          Regular expression constructs starting with `\'.

0036: @node Incremental Search
0037: @section Incremental Search
0038: @cindex incremental search
0039: @cindex isearch
0040: 
0041:   The principal search command in Emacs is @dfn{incremental}: it
0042: begins searching as soon as you type the first character of the search
0043: string.  As you type in the search string, Emacs shows you where the
0044: string (as you have typed it so far) would be found.  When you have
0045: typed enough characters to identify the place you want, you can stop.
0046: Depending on what you plan to do next, you may or may not need to
0047: terminate the search explicitly with @key{RET}.
0048: 
0049: @table @kbd
0050: @item C-s
0051: Incremental search forward (@code{isearch-forward}).
0052: @item C-r
0053: Incremental search backward (@code{isearch-backward}).
0054: @end table
0055:

0058: * Repeat Isearch::       Searching for the same string again.
0059: * Isearch Yank::         Commands that grab text into the search string

0061: * Error in Isearch::     When your string is not found.
0062: * Special Isearch::      Special input in incremental search.

0067: @node Basic Isearch
0068: @subsection Basics of Incremental Search
0069: 
0070: @table @kbd
0071: @item C-s
0072: Begin incremental search (@code{isearch-forward}).
0073: @item C-r
0074: Begin reverse incremental search (@code{isearch-backward}).
0075: @end table
0076: 
0077: @kindex C-s
0078: @findex isearch-forward
0079:   @kbd{C-s} (@code{isearch-forward}) starts a forward incremental
0080: search.  It reads characters from the keyboard, and moves point just
0081: past the end of the next occurrence of those characters in the buffer.
0082: 
0083:   For instance, if you type @kbd{C-s} and then @kbd{F}, that puts the
0084: cursor after the first @samp{F} that occurs in the buffer after the
0085: starting point.  If you then type @kbd{O}, the cursor moves to just
0086: after the first @samp{FO}; the @samp{F} in that @samp{FO} might not be
0087: the first @samp{F} previously found.  After another @kbd{O}, the
0088: cursor moves to just after the first @samp{FOO}.
0089: 
0090: @cindex faces for highlighting search matches
0091: @cindex isearch face
0092:   At each step, Emacs highlights the @dfn{current match}---the buffer
0093: text that matches the search string---using the @code{isearch} face
0094: (@pxref{Faces}).  @xref{Search Customizations}, for various options
0095: that customize this highlighting.  The current search string is also
0096: displayed in the echo area.
0097: 
0098:   If you make a mistake typing the search string, type @key{DEL}.
0099: Each @key{DEL} cancels the last character of the search string.
0100: @xref{Error in Isearch}, for more about dealing with unsuccessful
0101: search.
0102: 
0103: @cindex exit incremental search
0104: @cindex incremental search, exiting
0105:   When you are satisfied with the place you have reached, type
0106: @key{RET}.  This stops searching, leaving the cursor where the search
0107: brought it.  Also, any command not specially meaningful in searches
0108: stops the searching and is then executed.  Thus, typing @kbd{C-a}
0109: exits the search and then moves to the beginning of the line; typing
0110: one of the arrow keys exits the search and performs the respective
0111: movement command; etc.  @key{RET} is necessary only if the next
0112: command you want to type is a printing character, @key{DEL},
0113: @key{RET}, or another character that is special within searches
0114: (@kbd{C-q}, @kbd{C-w}, @kbd{C-r}, @kbd{C-s}, @kbd{C-y}, @kbd{M-y},
0115: @kbd{M-r}, @kbd{M-c}, @kbd{M-e}, and some others described below).
0116: You can fine-tune the commands that exit the search; see @ref{Not
0117: Exiting Isearch}.
0118: 
0119:   As a special exception, entering @key{RET} when the search string is
0120: empty launches nonincremental search (@pxref{Nonincremental Search}).
0121: (This can be customized; see @ref{Search Customizations}.)
0122: 
0123:   To abandon the search and return to the place where you started,
0124: type @kbd{@key{ESC} @key{ESC} @key{ESC}} (@code{isearch-cancel}) or
0125: @kbd{C-g C-g} (@code{isearch-abort}).
0126: 
0127:   When you exit the incremental search, it adds the original value of
0128: point to the mark ring, without activating the mark; you can thus use
0129: @kbd{C-u C-@key{SPC}} or @kbd{C-x C-x} to return to where you were
0130: before beginning the search.  @xref{Mark Ring}.  (Emacs only does this
0131: if the mark was not already active; if the mark was active when you
0132: started the search, both @kbd{C-u C-@key{SPC}} and @kbd{C-x C-x} will
0133: go to the mark.)
0134: 
0135: @kindex C-r
0136: @findex isearch-backward
0137:   To search backwards, use @kbd{C-r} (@code{isearch-backward}) instead
0138: of @kbd{C-s} to start the search.  A backward search finds matches
0139: that end before the starting point, just as a forward search finds
0140: matches that begin after it.
0141: 
0142: @node Repeat Isearch
0143: @subsection Repeating Incremental Search
0144: 
0145:   Suppose you search forward for @samp{FOO} and find a match, but not
0146: the one you expected to find: the @samp{FOO} you were aiming for
0147: occurs later in the buffer.  In this event, type another @kbd{C-s} to
0148: move to the next occurrence of the search string.  You can repeat this
0149: any number of times.  If you overshoot, you can cancel some @kbd{C-s}
0150: characters with @key{DEL}.  Similarly, each @kbd{C-r} in a backward
0151: incremental search repeats the backward search.
0152: 
0153: @cindex lazy search highlighting
0154:   If you pause for a little while during incremental search, Emacs
0155: highlights all the other possible matches for the search string that
0156: are present on the screen.  This helps you anticipate where you can
0157: get to by typing @kbd{C-s} or @kbd{C-r} to repeat the search.  The
0158: other matches are highlighted differently from the current match,
0159: using the customizable face @code{lazy-highlight} (@pxref{Faces}).  If
0160: you don't like this feature, you can disable it by setting
0161: @code{isearch-lazy-highlight} to @code{nil}.  For other customizations
0162: related to highlighting matches, see @ref{Search Customizations}.
0163: 
0164:   After exiting a search, you can search for the same string again by
0165: typing just @kbd{C-s C-s}.  The first @kbd{C-s} is the key that
0166: invokes incremental search, and the second @kbd{C-s} means to search
0167: again for the last search string.  Similarly, @kbd{C-r C-r} searches
0168: backward for the last search string.  In determining the last search
0169: string, it doesn't matter whether that string was searched for with
0170: @kbd{C-s} or @kbd{C-r}.
0171: 
0172:   If you are searching forward but you realize you were looking for
0173: something before the starting point, type @kbd{C-r} to switch to a
0174: backward search, leaving the search string unchanged.  Similarly,
0175: @kbd{C-s} in a backward search switches to a forward search.
0176: 
0177: @cindex search, wrapping around
0178: @cindex search, overwrapped
0179: @cindex wrapped search
0180: @cindex overwrapped search
0181:   If a search is failing and you ask to repeat it by typing another
0182: @kbd{C-s}, it starts again from the beginning of the buffer.
0183: Repeating a failing reverse search with @kbd{C-r} starts again from
0184: the end.  This is called @dfn{wrapping around}, and @samp{Wrapped}
0185: appears in the search prompt once this has happened.  If you keep on
0186: going past the original starting point of the search, it changes to
0187: @samp{Overwrapped}, which means that you are revisiting matches that
0188: you have already seen.
0189: 
0190: @cindex search ring
0191: @kindex M-n @r{(Incremental search)}
0192: @kindex M-p @r{(Incremental search)}
0193: @vindex search-ring-max
0194:   To reuse earlier search strings, use the @dfn{search ring}.  The
0195: commands @kbd{M-p} and @kbd{M-n} move through the ring to pick a
0196: search string to reuse.  These commands leave the selected search ring
0197: element in the minibuffer, where you can edit it.  Type
0198: @kbd{C-s}/@kbd{C-r} or @key{RET} to accept the string and start
0199: searching for it.  The number of most recently used search strings
0200: saved in the search ring is specified by the variable
0201: @code{search-ring-max}, 16 by default.
0202: 
0203: @cindex incremental search, edit search string
0204: @cindex interactively edit search string
0205: @kindex M-e @r{(Incremental search)}
0206: @kindex mouse-1 @r{in the minibuffer (Incremental Search)}
0207:   To edit the current search string in the minibuffer without
0208: replacing it with items from the search ring, type @kbd{M-e} or click
0209: @kbd{mouse-1} in the minibuffer.  Type @key{RET}, @kbd{C-s} or
0210: @kbd{C-r} to finish editing the string and search for it.  Type
0211: @kbd{C-f} or @kbd{@key{RIGHT}} to add to the search string characters
0212: following point from the buffer from which you started the search.

0214: @node Isearch Yank
0215: @subsection Isearch Yanking
0216: 
0217:   In many cases, you will want to use text at or near point as your
0218: search string.  The commands described in this subsection let you do
0219: that conveniently.
0220: 
0221: @kindex C-w @r{(Incremental search)}
0222: @findex isearch-yank-word-or-char
0223:   @kbd{C-w} (@code{isearch-yank-word-or-char}) appends the next
0224: character or word at point to the search string.  This is an easy way
0225: to search for another occurrence of the text at point.  (The decision
0226: of whether to copy a character or a word is heuristic.)
0227: 
0228: @kindex M-s C-e @r{(Incremental search)}
0229: @findex isearch-yank-line
0230:   Similarly, @kbd{M-s C-e} (@code{isearch-yank-line}) appends the rest
0231: of the current line to the search string.  If point is already at the
0232: end of a line, it appends the next line.  With a prefix argument
0233: @var{n}, it appends the next @var{n} lines.
0234: 
0235: @kindex C-y @r{(Incremental search)}
0236: @kindex M-y @r{(Incremental search)}
0237: @kindex mouse-2 @r{in the minibuffer (Incremental search)}
0238: @findex isearch-yank-kill
0239: @findex isearch-yank-pop
0240: @findex isearch-yank-x-selection
0241:   Within incremental search, @kbd{C-y} (@code{isearch-yank-kill})
0242: appends the current kill to the search string.  @kbd{M-y}
0243: (@code{isearch-yank-pop}), if called after @kbd{C-y}, replaces that
0244: appended text with an earlier kill, similar to the usual @kbd{M-y}
0245: (@code{yank-pop}) command (@pxref{Yanking}).  Clicking @kbd{mouse-2}
0246: in the echo area appends the current X selection (@pxref{Primary
0247: Selection}) to the search string (@code{isearch-yank-x-selection}).
0248: 
0249: @kindex C-M-w @r{(Incremental search)}
0250: @kindex C-M-y @r{(Incremental search)}
0251: @findex isearch-del-char
0252: @findex isearch-yank-char
0253:   @kbd{C-M-w} (@code{isearch-del-char}) deletes the last character
0254: from the search string, and @kbd{C-M-y} (@code{isearch-yank-char})
0255: appends the character after point to the search string.  An
0256: alternative method to add the character after point is to enter the
0257: minibuffer with @kbd{M-e} (@pxref{Repeat Isearch}) and type @kbd{C-f}
0258: or @kbd{@key{RIGHT}} at the end of the search string in the
0259: minibuffer.  Each @kbd{C-f} or @kbd{@key{RIGHT}} you type adds another
0260: character following point to the search string.
0261: 
0262:   Normally, when the search is case-insensitive, text yanked into the
0263: search string is converted to lower case, so that the search remains
0264: case-insensitive (@pxref{Lax Search, case folding}).  However, if the
0265: value of the variable @code{search-upper-case} (@pxref{Lax Search,
0266: search-upper-case}) is other than @code{not-yanks}, that disables this
0267: down-casing.

0269: @node Error in Isearch
0270: @subsection Errors in Incremental Search
0271: 
0272: @cindex isearch-fail face
0273:   If your string is not found at all, the echo area says @samp{Failing
0274: I-Search}, and the cursor moves past the place where Emacs found as
0275: much of your string as it could.  Thus, if you search for @samp{FOOT},
0276: and there is no @samp{FOOT}, you might see the cursor after the
0277: @samp{FOO} in @samp{FOOL}.  In the echo area, the part of the search
0278: string that failed to match is highlighted using the face
0279: @code{isearch-fail}.
0280: 
0281:   At this point, there are several things you can do.  If your string
0282: was mistyped, you can use @key{DEL} to erase some of it and correct
0283: it, or you can type @kbd{M-e} and edit it.  If you like the place you
0284: have found, you can type @key{RET} to remain there.  Or you can type
0285: @kbd{C-g}, which removes from the search string the characters that
0286: could not be found (the @samp{T} in @samp{FOOT}), leaving those that
0287: were found (the @samp{FOO} in @samp{FOOT}).  A second @kbd{C-g} at
0288: that point cancels the search entirely, returning point to where it
0289: was when the search started.
0290: 
0291: @cindex quitting (in search)
0292: @kindex C-g @r{(Incremental search)}
0293:   The quit command, @kbd{C-g}, does special things during searches;
0294: just what it does depends on the status of the search.  If the search
0295: has found what you specified and is waiting for input, @kbd{C-g}
0296: cancels the entire search, moving the cursor back to where you started
0297: the search.  If @kbd{C-g} is typed when there are characters in the
0298: search string that have not been found---because Emacs is still
0299: searching for them, or because it has failed to find them---then the
0300: search string characters which have not been found are discarded from
0301: the search string.  With them gone, the search is now successful and
0302: waiting for more input, so a second @kbd{C-g} will cancel the entire
0303: search.

0424: @node Not Exiting Isearch
0425: @subsection Not Exiting Incremental Search
0426: 
0427: This subsection describes how to control whether typing a command not
0428: specifically meaningful is searches exits the search before executing
0429: the command.  It also describes two categories of commands which you
0430: can type without exiting the current incremental search, even though
0431: they are not themselves part of incremental search.
0432: 
0433: @vindex search-exit-option
0434:   Normally, typing a command that is not bound by the incremental
0435: search exits the search before executing the command.  Thus, the
0436: command operates on the buffer from which you invoked the search.
0437: However, if you customize the variable @code{search-exit-option} to
0438: @code{nil}, the characters which you type that are not interpreted by
0439: the incremental search are simply appended to the search string.  This
0440: is so you could include in the search string control characters, such
0441: as @kbd{C-a}, that would normally exit the search and invoke the
0442: command bound to them on the buffer.
0443: 
0444: @table @asis
0445: @item Prefix Arguments
0446: @cindex prefix argument commands, during incremental search
0447: @vindex isearch-allow-prefix
0448:   In incremental search, when you type a command that specifies a
0449: prefix argument (@pxref{Arguments}), by default it will apply either
0450: to the next action in the search or to the command that exits the
0451: search.  In other words, entering a prefix argument will not by itself
0452: terminate the search.
0453: 
0454:   In previous versions of Emacs, entering a prefix argument always
0455: terminated the search.  You can revert to this behavior by setting the
0456: variable @code{isearch-allow-prefix} to @code{nil}.
0457: 
0458:   When @code{isearch-allow-scroll} is non-@code{nil} (see below),
0459: prefix arguments always have the default behavior described above,
0460: i.e., they don't terminate the search, even if
0461: @code{isearch-allow-prefix} is @code{nil}.
0462: 
0463: @item Scrolling Commands
0464: @cindex scrolling commands, during incremental search
0465: @vindex isearch-allow-scroll
0466:   Normally, scrolling commands exit incremental search.  If you change
0467: the variable @code{isearch-allow-scroll} to a non-@code{nil} value,
0468: that enables the use of the scroll-bar, as well as keyboard scrolling
0469: commands like @kbd{C-v}, @kbd{M-v}, and @kbd{C-l} (@pxref{Scrolling}).
0470: This applies only to calling these commands via their bound key
0471: sequences---typing @kbd{M-x} will still exit the search.  You can give
0472: prefix arguments to these commands in the usual way.  This feature
0473: won't let you scroll the current match out of visibility, however.
0474: 
0475:   The @code{isearch-allow-scroll} feature also affects some other
0476: commands, such as @kbd{C-x 2} (@code{split-window-below}) and

0478: affect where the text appears on the screen.  It applies to any
0479: command whose name has a non-@code{nil} @code{isearch-scroll}
0480: property.  So you can control which commands are affected by changing
0481: these properties.

0491: @end example

0501: @node Isearch Minibuffer
0502: @subsection Searching the Minibuffer
0503: @cindex minibuffer history, searching
0504: 
0505: If you start an incremental search while the minibuffer is active,
0506: Emacs searches the contents of the minibuffer.  Unlike searching an
0507: ordinary buffer, the search string is not shown in the echo area,
0508: because that is used to display the minibuffer.
0509: 
0510: If an incremental search fails in the minibuffer, it tries searching
0511: the minibuffer history.  @xref{Minibuffer History}.  You can visualize
0512: the minibuffer and its history as a series of pages, with the
0513: earliest history element on the first page and the current minibuffer
0514: on the last page.  A forward search, @kbd{C-s}, searches forward to
0515: later pages; a reverse search, @kbd{C-r}, searches backwards to
0516: earlier pages.  Like in ordinary buffer search, a failing search can
0517: wrap around, going from the last page to the first page or vice versa.
0518: 
0519: When the current match is on a history element, that history element
0520: is pulled into the minibuffer.  If you exit the incremental search
0521: normally (e.g., by typing @key{RET}), it remains in the minibuffer
0522: afterwards.  Canceling the search, with @kbd{C-g}, restores the
0523: contents of the minibuffer when you began the search.

0526: @section Nonincremental Search
0527: @cindex nonincremental search
0528: 
0529:   Emacs also has conventional nonincremental search commands, which require
0530: you to type the entire search string before searching begins.
0531: 
0532: @table @kbd
0533: @item C-s @key{RET} @var{string} @key{RET}
0534: Search for @var{string}.
0535: @item C-r @key{RET} @var{string} @key{RET}
0536: Search backward for @var{string}.
0537: @end table
0538: 
0539:   To start a nonincremental search, first type @kbd{C-s @key{RET}}.
0540: This enters the minibuffer to read the search string; terminate the
0541: string with @key{RET}, and then the search takes place.  If the string
0542: is not found, the search command signals an error.
0543: 
0544:   When you type @kbd{C-s @key{RET}}, the @kbd{C-s} invokes incremental
0545: search as usual.  That command is specially programmed to invoke the
0546: command for nonincremental search, if the string you specify is empty.
0547: (Such an empty argument would otherwise be useless.)  @kbd{C-r
0548: @key{RET}} does likewise, invoking the nonincremental
0549: backward-searching command.
0550: 
0551:   Nonincremental search can also be invoked form the menu bar's
0552: @samp{Edit->Search} menu.
0553: 
0554: @findex search-forward
0555: @findex search-backward
0556:   You can also use two simpler commands, @kbd{M-x search-forward} and
0557: @kbd{M-x search-backward}.  These commands look for the literal
0558: strings you specify, and don't support any of the lax-search features
0559: (@pxref{Lax Search}) except case folding.

0561: @node Word Search
0562: @section Word Search
0563: @cindex word search
0564: 
0565:   A @dfn{word search} finds a sequence of words without regard to the
0566: type of punctuation between them.  For instance, if you enter a search
0567: string that consists of two words separated by a single space, the
0568: search matches any sequence of those two words separated by one or
0569: more spaces, newlines, or other punctuation characters.  This is
0570: particularly useful for searching text documents, because you don't
0571: have to worry whether the words you are looking for are separated by
0572: newlines or spaces.  Note that major modes for programming languages
0573: or other specialized modes can modify the definition of a word to suit
0574: their syntactic needs.
0575: 
0576: @table @kbd
0577: @item M-s w
0578:   If incremental search is active, toggle word search mode
0579: (@code{isearch-toggle-word}); otherwise, begin an incremental forward
0580: word search (@code{isearch-forward-word}).
0581: @item M-s w @key{RET} @var{words} @key{RET}
0582: Search for @var{words}, using a forward nonincremental word search.
0583: @item M-s w C-r @key{RET} @var{words} @key{RET}
0584: Search backward for @var{words}, using a nonincremental word search.
0585: @item M-s M-w
0586: Search the Web for the text in region.
0587: @end table
0588: 
0589: @kindex M-s w
0590: @findex isearch-forward-word
0591:   To begin a forward incremental word search, type @kbd{M-s w}.  If
0592: incremental search is not already active, this runs the command
0593: @code{isearch-forward-word}.  If incremental search is already active
0594: (whether a forward or backward search), @kbd{M-s w} switches to a word
0595: search while keeping the direction of the search and the current
0596: search string unchanged.  You can toggle word search back off by
0597: typing @kbd{M-s w} again.
0598: 
0599: @findex word-search-forward
0600: @findex word-search-backward
0601:   To begin a nonincremental word search, type @kbd{M-s w @key{RET}}
0602: for a forward search, or @kbd{M-s w C-r @key{RET}} for a backward search.
0603: These run the commands @code{word-search-forward} and
0604: @code{word-search-backward} respectively.
0605: 
0606:   Incremental and nonincremental word searches differ slightly in the
0607: way they find a match.  In a nonincremental word search, each word in
0608: the search string must exactly match a whole word.  In an incremental
0609: word search, the matching is more lax: while you are typing the search
0610: string, its first and last words need not match whole words.  This is
0611: so that the matching can proceed incrementally as you type.  This
0612: additional laxity does not apply to the lazy highlight
0613: (@pxref{Incremental Search}), which always matches whole words.
0614: While you are typing the search string, @samp{Pending} appears in the
0615: search prompt until you use a search repeating key like @kbd{C-s}.
0616: 
0617:   The word search commands don't perform character folding, and
0618: toggling lax whitespace matching (@pxref{Lax Search, lax space
0619: matching}) has no effect on them.
0620: 
0621: @kindex M-s M-w
0622: @findex eww-search-words
0623: @vindex eww-search-prefix
0624:   To search the Web for the text in region, type @kbd{M-s M-w}.  This
0625: command performs an Internet search for the words in region using the
0626: search engine whose @acronym{URL} is specified by the variable
0627: @code{eww-search-prefix} (@pxref{Basics, EWW, , eww, The Emacs Web
0628: Wowser Manual}).  If the region is not active, or doesn't contain any
0629: words, this command prompts the user for a URL or keywords to search.

0686: @node Regexp Search
0687: @section Regular Expression Search
0688: @cindex regexp search
0689: @cindex search for a regular expression
0690: 
0691:   A @dfn{regular expression} (or @dfn{regexp} for short) is a pattern
0692: that denotes a class of alternative strings to match.  Emacs
0693: provides both incremental and nonincremental ways to search for a
0694: match for a regexp.  The syntax of regular expressions is explained in
0695: the next section.
0696: 
0697: @table @kbd
0698: @item C-M-s
0699: Begin incremental regexp search (@code{isearch-forward-regexp}).
0700: @item C-M-r
0701: Begin reverse incremental regexp search (@code{isearch-backward-regexp}).
0702: @end table
0703: 
0704: @kindex C-M-s
0705: @findex isearch-forward-regexp
0706: @kindex C-M-r
0707: @findex isearch-backward-regexp
0708:   Incremental search for a regexp is done by typing @kbd{C-M-s}
0709: (@code{isearch-forward-regexp}), by invoking @kbd{C-s} with a
0710: prefix argument (whose value does not matter), or by typing @kbd{M-r}
0711: within a forward incremental search.  This command reads a
0712: search string incrementally just like @kbd{C-s}, but it treats the
0713: search string as a regexp rather than looking for an exact match
0714: against the text in the buffer.  Each time you add text to the search
0715: string, you make the regexp longer, and the new regexp is searched
0716: for.  To search backward for a regexp, use @kbd{C-M-r}
0717: (@code{isearch-backward-regexp}), @kbd{C-r} with a prefix argument,
0718: or @kbd{M-r} within a backward incremental search.
0719: 
0720: @vindex regexp-search-ring-max
0721:   All of the special key sequences in an ordinary incremental search
0722: (@pxref{Special Isearch}) do similar things in an incremental regexp
0723: search.  For instance, typing @kbd{C-s} immediately after starting the
0724: search retrieves the last incremental search regexp used and searches
0725: forward for it.  Incremental regexp and non-regexp searches have
0726: independent defaults.  They also have separate search rings, which you
0727: can access with @kbd{M-p} and @kbd{M-n}.  The maximum number of search
0728: regexps saved in the search ring is determined by the value of
0729: @code{regexp-search-ring-max}, 16 by default.
0730: 
0731:   Unlike ordinary incremental search, incremental regexp search
0732: does not use lax space matching by default.  To toggle this feature
0733: use @kbd{M-s @key{SPC}} (@code{isearch-toggle-lax-whitespace}).
0734: Then any @key{SPC} typed in incremental regexp search will match
0735: any sequence of one or more whitespace characters.  The variable
0736: @code{search-whitespace-regexp} specifies the regexp for the lax
0737: space matching.  @xref{Special Isearch}.
0738: 
0739:   Also unlike ordinary incremental search, incremental regexp search
0740: cannot use character folding (@pxref{Lax Search}).  (If you toggle
0741: character folding during incremental regexp search with @kbd{M-s '},
0742: the search becomes a non-regexp search and the search pattern you
0743: typed is interpreted as a literal string.)
0744: 
0745:   In some cases, adding characters to the regexp in an incremental
0746: regexp search can make the cursor move back and start again.  For
0747: example, if you have searched for @samp{foo} and you add @samp{\|bar},
0748: the cursor backs up in case the first @samp{bar} precedes the first
0749: @samp{foo}.  @xref{Regexps}.
0750: 
0751:   Forward and backward regexp search are not symmetrical, because
0752: regexp matching in Emacs always operates forward, starting with the
0753: beginning of the regexp.  Thus, forward regexp search scans forward,
0754: trying a forward match at each possible starting position.  Backward
0755: regexp search scans backward, trying a forward match at each possible
0756: starting position.  These search methods are not mirror images.
0757: 
0758: @findex re-search-forward
0759: @findex re-search-backward
0760:   Nonincremental search for a regexp is done with the commands
0761: @code{re-search-forward} and @code{re-search-backward}.  You can
0762: invoke these with @kbd{M-x}, or by way of incremental regexp search
0763: with @kbd{C-M-s @key{RET}} and @kbd{C-M-r @key{RET}}.  When you invoke
0764: these commands with @kbd{M-x}, they search for the exact regexp you
0765: specify, and thus don't support any lax-search features (@pxref{Lax
0766: Search}) except case folding.
0767: 
0768:   If you use the incremental regexp search commands with a prefix
0769: argument, they perform ordinary string search, like
0770: @code{isearch-forward} and @code{isearch-backward}.  @xref{Incremental
0771: Search}.
0772: 
0773: @node Regexps
0774: @section Syntax of Regular Expressions
0775: @cindex syntax of regexps
0776: @cindex regular expression
0777: @cindex regexp
0778: 
0779:   This manual describes regular expression features that users
0780: typically use.  @xref{Regular Expressions,,, elisp, The Emacs Lisp
0781: Reference Manual}, for additional features used mainly in Lisp
0782: programs.
0783: 
0784:   Regular expressions have a syntax in which a few characters are
0785: special constructs and the rest are @dfn{ordinary}.  An ordinary
0786: character matches that same character and nothing else.  The special
0787: characters are @samp{$^.*+?[\}.  The character @samp{]} is special if
0788: it ends a character alternative (see later).  The character @samp{-}
0789: is special inside a character alternative.  Any other character
0790: appearing in a regular expression is ordinary, unless a @samp{\}
0791: precedes it.  (When you use regular expressions in a Lisp program,
0792: each @samp{\} must be doubled, see the example near the end of this
0793: section.)
0794: 
0795:   For example, @samp{f} is not a special character, so it is ordinary, and
0796: therefore @samp{f} is a regular expression that matches the string
0797: @samp{f} and no other string.  (It does @emph{not} match the string
0798: @samp{ff}.)  Likewise, @samp{o} is a regular expression that matches
0799: only @samp{o}.  (When case distinctions are being ignored, these regexps
0800: also match @samp{F} and @samp{O}, but we consider this a generalization
0801: of ``the same string'', rather than an exception.)
0802: 
0803:   Any two regular expressions @var{a} and @var{b} can be concatenated.
0804: The result is a regular expression which matches a string if @var{a}
0805: matches some amount of the beginning of that string and @var{b}
0806: matches the rest of the string.  For example, concatenating the
0807: regular expressions @samp{f} and @samp{o} gives the regular expression
0808: @samp{fo}, which matches only the string @samp{fo}.  Still trivial.
0809: To do something nontrivial, you need to use one of the special
0810: characters.  Here is a list of them.
0811: 
0812: @table @asis
0813: @item @kbd{.}@: @r{(Period)}
0814: is a special character that matches any single character except a
0815: newline.  For example, the regular expressions @samp{a.b} matches any
0816: three-character string that begins with @samp{a} and ends with
0817: @samp{b}.
0818: 
0819: @item @kbd{*}
0820: is not a construct by itself; it is a postfix operator that means to
0821: match the preceding regular expression repetitively any number of
0822: times, as many times as possible.  Thus, @samp{o*} matches any number
0823: of @samp{o}s, including no @samp{o}s.
0824: 
0825: @samp{*} always applies to the @emph{smallest} possible preceding
0826: expression.  Thus, @samp{fo*} has a repeating @samp{o}, not a repeating
0827: @samp{fo}.  It matches @samp{f}, @samp{fo}, @samp{foo}, and so on.
0828: 
0829: The matcher processes a @samp{*} construct by matching, immediately,
0830: as many repetitions as can be found.  Then it continues with the rest
0831: of the pattern.  If that fails, backtracking occurs, discarding some
0832: of the matches of the @samp{*}-modified construct in case that makes
0833: it possible to match the rest of the pattern.  For example, in matching
0834: @samp{ca*ar} against the string @samp{caaar}, the @samp{a*} first
0835: tries to match all three @samp{a}s; but the rest of the pattern is
0836: @samp{ar} and there is only @samp{r} left to match, so this try fails.
0837: The next alternative is for @samp{a*} to match only two @samp{a}s.
0838: With this choice, the rest of the regexp matches successfully.
0839: 
0840: @item @kbd{+}
0841: is a postfix operator, similar to @samp{*} except that it must match
0842: the preceding expression at least once.  Thus, @samp{ca+r} matches the
0843: strings @samp{car} and @samp{caaaar} but not the string @samp{cr},
0844: whereas @samp{ca*r} matches all three strings.
0845: 
0846: @item @kbd{?}
0847: is a postfix operator, similar to @samp{*} except that it can match
0848: the preceding expression either once or not at all.  Thus, @samp{ca?r}
0849: matches @samp{car} or @samp{cr}, and nothing else.
0850: 
0851: @item @kbd{*?}, @kbd{+?}, @kbd{??}
0852: @cindex non-greedy regexp matching
0853: are non-@dfn{greedy} variants of the operators above.  The normal
0854: operators @samp{*}, @samp{+}, @samp{?} match as much as they can, as
0855: long as the overall regexp can still match.  With a following
0856: @samp{?}, they will match as little as possible.
0857: 
0858: Thus, both @samp{ab*} and @samp{ab*?} can match the string @samp{a}
0859: and the string @samp{abbbb}; but if you try to match them both against
0860: the text @samp{abbb}, @samp{ab*} will match it all (the longest valid
0861: match), while @samp{ab*?}  will match just @samp{a} (the shortest
0862: valid match).
0863: 
0864: Non-greedy operators match the shortest possible string starting at a
0865: given starting point; in a forward search, though, the earliest
0866: possible starting point for match is always the one chosen.  Thus, if
0867: you search for @samp{a.*?$} against the text @samp{abbab} followed by
0868: a newline, it matches the whole string.  Since it @emph{can} match
0869: starting at the first @samp{a}, it does.
0870: 
0871: @item @kbd{\@{@var{n}\@}}
0872: is a postfix operator specifying @var{n} repetitions---that is, the
0873: preceding regular expression must match exactly @var{n} times in a
0874: row.  For example, @samp{x\@{4\@}} matches the string @samp{xxxx} and
0875: nothing else.
0876: 
0877: @item @kbd{\@{@var{n},@var{m}\@}}
0878: is a postfix operator specifying between @var{n} and @var{m}
0879: repetitions---that is, the preceding regular expression must match at
0880: least @var{n} times, but no more than @var{m} times.  If @var{m} is
0881: omitted, then there is no upper limit, but the preceding regular
0882: expression must match at least @var{n} times.@* @samp{\@{0,1\@}} is
0883: equivalent to @samp{?}. @* @samp{\@{0,\@}} is equivalent to
0884: @samp{*}. @* @samp{\@{1,\@}} is equivalent to @samp{+}.
0885: 
0886: @item @kbd{[ @dots{} ]}
0887: is a @dfn{character set}, beginning with @samp{[} and terminated by
0888: @samp{]}.
0889: 
0890: In the simplest case, the characters between the two brackets are what
0891: this set can match.  Thus, @samp{[ad]} matches either one @samp{a} or
0892: one @samp{d}, and @samp{[ad]*} matches any string composed of just
0893: @samp{a}s and @samp{d}s (including the empty string).  It follows that
0894: @samp{c[ad]*r} matches @samp{cr}, @samp{car}, @samp{cdr},
0895: @samp{caddaar}, etc.
0896: 
0897: You can also include character ranges in a character set, by writing the
0898: starting and ending characters with a @samp{-} between them.  Thus,
0899: @samp{[a-z]} matches any lower-case @acronym{ASCII} letter.  Ranges may be
0900: intermixed freely with individual characters, as in @samp{[a-z$%.]},
0901: which matches any lower-case @acronym{ASCII} letter or @samp{$}, @samp{%} or
0902: period.
0903: 
0904: You can also include certain special @dfn{character classes} in a
0905: character set.  A @samp{[:} and balancing @samp{:]} enclose a
0906: character class inside a character alternative.  For instance,
0907: @samp{[[:alnum:]]} matches any letter or digit.  @xref{Char Classes,,,
0908: elisp, The Emacs Lisp Reference Manual}, for a list of character
0909: classes.
0910: 
0911: To include a @samp{]} in a character set, you must make it the first
0912: character.  For example, @samp{[]a]} matches @samp{]} or @samp{a}.  To
0913: include a @samp{-}, write @samp{-} as the first or last character of the
0914: set, or put it after a range.  Thus, @samp{[]-]} matches both @samp{]}
0915: and @samp{-}.
0916: 
0917: To include @samp{^} in a set, put it anywhere but at the beginning of
0918: the set.  (At the beginning, it complements the set---see below.)
0919: 
0920: When you use a range in case-insensitive search, you should write both
0921: ends of the range in upper case, or both in lower case, or both should
0922: be non-letters.  The behavior of a mixed-case range such as @samp{A-z}
0923: is somewhat ill-defined, and it may change in future Emacs versions.
0924: 
0925: @item @kbd{[^ @dots{} ]}
0926: @samp{[^} begins a @dfn{complemented character set}, which matches any
0927: character except the ones specified.  Thus, @samp{[^a-z0-9A-Z]} matches
0928: all characters @emph{except} @acronym{ASCII} letters and digits.
0929: 
0930: @samp{^} is not special in a character set unless it is the first
0931: character.  The character following the @samp{^} is treated as if it
0932: were first (in other words, @samp{-} and @samp{]} are not special there).
0933: 
0934: A complemented character set can match a newline, unless newline is
0935: mentioned as one of the characters not to match.  This is in contrast to
0936: the handling of regexps in programs such as @code{grep}.
0937: 
0938: @item @kbd{^}
0939: is a special character that matches the empty string, but only at the
0940: beginning of a line in the text being matched.  Otherwise it fails to
0941: match anything.  Thus, @samp{^foo} matches a @samp{foo} that occurs at
0942: the beginning of a line.
0943: 
0944: For historical compatibility reasons, @samp{^} can be used with this
0945: meaning only at the beginning of the regular expression, or after
0946: @samp{\(} or @samp{\|}.
0947: 
0948: @item @kbd{$}
0949: is similar to @samp{^} but matches only at the end of a line.  Thus,
0950: @samp{x+$} matches a string of one @samp{x} or more at the end of a line.
0951: 
0952: For historical compatibility reasons, @samp{$} can be used with this
0953: meaning only at the end of the regular expression, or before @samp{\)}
0954: or @samp{\|}.
0955: 
0956: @item @kbd{\}
0957: has two functions: it quotes the special characters (including
0958: @samp{\}), and it introduces additional special constructs.
0959: 
0960: Because @samp{\} quotes special characters, @samp{\$} is a regular
0961: expression that matches only @samp{$}, and @samp{\[} is a regular
0962: expression that matches only @samp{[}, and so on.
0963: 
0964: See the following section for the special constructs that begin
0965: with @samp{\}.
0966: @end table
0967: 
0968:   Note: for historical compatibility, special characters are treated as
0969: ordinary ones if they are in contexts where their special meanings make no
0970: sense.  For example, @samp{*foo} treats @samp{*} as ordinary since there is
0971: no preceding expression on which the @samp{*} can act.  It is poor practice
0972: to depend on this behavior; it is better to quote the special character anyway,
0973: regardless of where it appears.
0974: 
0975: As a @samp{\} is not special inside a character alternative, it can
0976: never remove the special meaning of @samp{-} or @samp{]}.  So you
0977: should not quote these characters when they have no special meaning
0978: either.  This would not clarify anything, since backslashes can
0979: legitimately precede these characters where they @emph{have} special
0980: meaning, as in @samp{[^\]} (@code{"[^\\]"} for Lisp string syntax),
0981: which matches any single character except a backslash.

0984: @section Backslash in Regular Expressions
0985: 
0986:   For the most part, @samp{\} followed by any character matches only
0987: that character.  However, there are several exceptions: two-character
0988: sequences starting with @samp{\} that have special meanings.  The
0989: second character in the sequence is always an ordinary character when
0990: used on its own.  Here is a table of @samp{\} constructs.
0991: 
0992: @table @kbd
0993: @item \|
0994: specifies an alternative.  Two regular expressions @var{a} and @var{b}
0995: with @samp{\|} in between form an expression that matches some text if
0996: either @var{a} matches it or @var{b} matches it.  It works by trying to
0997: match @var{a}, and if that fails, by trying to match @var{b}.
0998: 
0999: Thus, @samp{foo\|bar} matches either @samp{foo} or @samp{bar}
1000: but no other string.
1001: 
1002: @samp{\|} applies to the largest possible surrounding expressions.  Only a
1003: surrounding @samp{\( @dots{} \)} grouping can limit the grouping power of
1004: @samp{\|}.
1005: 
1006: Full backtracking capability exists to handle multiple uses of @samp{\|}.
1007: 
1008: @item \( @dots{} \)
1009: is a grouping construct that serves three purposes:
1010: 
1011: @enumerate
1012: @item
1013: To enclose a set of @samp{\|} alternatives for other operations.
1014: Thus, @samp{\(foo\|bar\)x} matches either @samp{foox} or @samp{barx}.
1015: 
1016: @item
1017: To enclose a complicated expression for the postfix operators @samp{*},
1018: @samp{+} and @samp{?} to operate on.  Thus, @samp{ba\(na\)*} matches
1019: @samp{bananana}, etc., with any (zero or more) number of @samp{na}
1020: strings.
1021: 
1022: @item
1023: To record a matched substring for future reference.
1024: @end enumerate
1025: 
1026: This last application is not a consequence of the idea of a
1027: parenthetical grouping; it is a separate feature that is assigned as a
1028: second meaning to the same @samp{\( @dots{} \)} construct.  In practice
1029: there is usually no conflict between the two meanings; when there is
1030: a conflict, you can use a shy group.
1031: 
1032: @item \(?: @dots{} \)
1033: @cindex shy group, in regexp
1034: specifies a shy group that does not record the matched substring;
1035: you can't refer back to it with @samp{\@var{d}} (see below).  This is
1036: useful in mechanically combining regular expressions, so that you can
1037: add groups for syntactic purposes without interfering with the
1038: numbering of the groups that are meant to be referred to.
1039: 
1040: @item \@var{d}
1041: @cindex back reference, in regexp
1042: matches the same text that matched the @var{d}th occurrence of a
1043: @samp{\( @dots{} \)} construct.  This is called a @dfn{back
1044: reference}.
1045: 
1046: After the end of a @samp{\( @dots{} \)} construct, the matcher remembers
1047: the beginning and end of the text matched by that construct.  Then,
1048: later on in the regular expression, you can use @samp{\} followed by the
1049: digit @var{d} to mean ``match the same text matched the @var{d}th time
1050: by the @samp{\( @dots{} \)} construct''.
1051: 
1052: The strings matching the first nine @samp{\( @dots{} \)} constructs
1053: appearing in a regular expression are assigned numbers 1 through 9 in
1054: the order that the open-parentheses appear in the regular expression.
1055: So you can use @samp{\1} through @samp{\9} to refer to the text matched
1056: by the corresponding @samp{\( @dots{} \)} constructs.
1057: 
1058: For example, @samp{\(.*\)\1} matches any newline-free string that is
1059: composed of two identical halves.  The @samp{\(.*\)} matches the first
1060: half, which may be anything, but the @samp{\1} that follows must match
1061: the same exact text.
1062: 
1063: If a particular @samp{\( @dots{} \)} construct matches more than once
1064: (which can easily happen if it is followed by @samp{*}), only the last
1065: match is recorded.
1066: 
1067: @item \`
1068: matches the empty string, but only at the beginning of the string or
1069: buffer (or its accessible portion) being matched against.
1070: 
1071: @item \'
1072: matches the empty string, but only at the end of the string or buffer
1073: (or its accessible portion) being matched against.
1074: 
1075: @item \=
1076: matches the empty string, but only at point.
1077: 
1078: @item \b
1079: matches the empty string, but only at the beginning or
1080: end of a word.  Thus, @samp{\bfoo\b} matches any occurrence of
1081: @samp{foo} as a separate word.  @samp{\bballs?\b} matches
1082: @samp{ball} or @samp{balls} as a separate word.
1083: 
1084: @samp{\b} matches at the beginning or end of the buffer
1085: regardless of what text appears next to it.
1086: 
1087: @item \B
1088: matches the empty string, but @emph{not} at the beginning or
1089: end of a word.
1090: 
1091: @item \<
1092: matches the empty string, but only at the beginning of a word.
1093: @samp{\<} matches at the beginning of the buffer only if a
1094: word-constituent character follows.
1095: 
1096: @item \>
1097: matches the empty string, but only at the end of a word.  @samp{\>}
1098: matches at the end of the buffer only if the contents end with a
1099: word-constituent character.
1100: 
1101: @item \w
1102: matches any word-constituent character.  The syntax table determines
1103: which characters these are.  @xref{Syntax Tables,, Syntax Tables,
1104: elisp, The Emacs Lisp Reference Manual}.
1105: 
1106: @item \W
1107: matches any character that is not a word-constituent.
1108: 
1109: @item \_<
1110: matches the empty string, but only at the beginning of a symbol.
1111: A symbol is a sequence of one or more symbol-constituent characters.
1112: A symbol-constituent character is a character whose syntax is either
1113: @samp{w} or @samp{_}.  @samp{\_<} matches at the beginning of the
1114: buffer only if a symbol-constituent character follows.
1115: 
1116: @item \_>
1117: matches the empty string, but only at the end of a symbol.  @samp{\_>}
1118: matches at the end of the buffer only if the contents end with a
1119: symbol-constituent character.
1120: 
1121: @item \s@var{c}
1122: matches any character whose syntax is @var{c}.  Here @var{c} is a
1123: character that designates a particular syntax class: thus, @samp{w}
1124: for word constituent, @samp{-} or @samp{ } for whitespace, @samp{.}
1125: for ordinary punctuation, etc.  @xref{Syntax Tables,, Syntax Tables,
1126: elisp, The Emacs Lisp Reference Manual}.
1127: 
1128: @item \S@var{c}
1129: matches any character whose syntax is not @var{c}.
1130: 
1131: @cindex categories of characters
1132: @cindex characters which belong to a specific language
1133: @findex describe-categories
1134: @item \c@var{c}
1135: matches any character that belongs to the category @var{c}.  For
1136: example, @samp{\cc} matches Chinese characters, @samp{\cg} matches
1137: Greek characters, etc.  For the description of the known categories,
1138: type @kbd{M-x describe-categories @key{RET}}.
1139: 
1140: @item \C@var{c}
1141: matches any character that does @emph{not} belong to category
1142: @var{c}.
1143: @end table
1144: 
1145:   The constructs that pertain to words and syntax are controlled by
1146: the setting of the syntax table.  @xref{Syntax Tables,, Syntax Tables,
1147: elisp, The Emacs Lisp Reference Manual}.

1167: @node Lax Search
1168: @section Lax Matching During Searching
1169: 
1170: @cindex lax search
1171: @cindex character equivalence in search
1172:   Normally, you'd want search commands to disregard certain minor
1173: differences between the search string you type and the text being
1174: searched.  For example, sequences of whitespace characters of
1175: different length are usually perceived as equivalent; letter-case
1176: differences usually don't matter; etc.  This is known as
1177: @dfn{character equivalence}.
1178: 
1179:   This section describes the Emacs lax search features, and how to
1180: tailor them to your needs.
1181: 
1182: @cindex lax space matching in search
1183: @kindex M-s SPC @r{(Incremental search)}
1184: @kindex SPC @r{(Incremental search)}
1185: @findex isearch-toggle-lax-whitespace
1186: @vindex search-whitespace-regexp
1187:   By default, search commands perform @dfn{lax space matching}:
1188: each space, or sequence of spaces, matches any sequence of one or more
1189: whitespace characters in the text.  (Incremental regexp search has a
1190: separate default; see @ref{Regexp Search}.)  Hence, @samp{foo bar}
1191: matches @samp{foo bar}, @samp{foo@w{  }bar}, @samp{foo@w{   }bar}, and
1192: so on (but not @samp{foobar}).  More precisely, Emacs matches each
1193: sequence of space characters in the search string to a regular
1194: expression specified by the variable @code{search-whitespace-regexp}.
1195: For example, to make spaces match sequences of newlines as well as
1196: spaces, set it to @samp{"[[:space:]\n]+"}.  The default value of this
1197: variable depends on the buffer's major mode; most major modes classify
1198: spaces, tabs, and formfeed characters as whitespace.
1199: 
1200:   If you want whitespace characters to match exactly, you can turn lax
1201: space matching off by typing @kbd{M-s @key{SPC}}
1202: (@code{isearch-toggle-lax-whitespace}) within an incremental search.
1203: Another @kbd{M-s @key{SPC}} turns lax space matching back on.  To
1204: disable lax whitespace matching for all searches, change
1205: @code{search-whitespace-regexp} to @code{nil}; then each space in the
1206: search string matches exactly one space.
1207: 
1208: @cindex case folding in search
1209: @cindex case-sensitivity and search
1210:   Searches in Emacs by default ignore the case of the text they are
1211: searching through, if you specify the search string in lower case.
1212: Thus, if you specify searching for @samp{foo}, then @samp{Foo} and
1213: @samp{foo} also match.  Regexps, and in particular character sets,
1214: behave likewise: @samp{[ab]} matches @samp{a} or @samp{A} or @samp{b}
1215: or @samp{B}.  This feature is known as @dfn{case folding}, and it is
1216: supported in both incremental and non-incremental search modes.
1217: 
1218: @vindex search-upper-case
1219:   An upper-case letter anywhere in the search string makes the search
1220: case-sensitive.  Thus, searching for @samp{Foo} does not find
1221: @samp{foo} or @samp{FOO}.  This applies to regular expression search
1222: as well as to literal string search.  The effect ceases if you delete
1223: the upper-case letter from the search string.  The variable
1224: @code{search-upper-case} controls this: if it is non-@code{nil} (the
1225: default), an upper-case character in the search string make the search
1226: case-sensitive; setting it to @code{nil} disables this effect of
1227: upper-case characters.
1228: 
1229: @vindex case-fold-search
1230:   If you set the variable @code{case-fold-search} to @code{nil}, then
1231: all letters must match exactly, including case.  This is a per-buffer
1232: variable; altering the variable normally affects only the current buffer,
1233: unless you change its default value.  @xref{Locals}.
1234: This variable applies to nonincremental searches also, including those
1235: performed by the replace commands (@pxref{Replace}) and the minibuffer
1236: history matching commands (@pxref{Minibuffer History}).
1237: 
1238: @kindex M-c @r{(Incremental search)}
1239: @kindex M-s c @r{(Incremental search)}
1240: @findex isearch-toggle-case-fold
1241:   Typing @kbd{M-c} or @kbd{M-s c} (@code{isearch-toggle-case-fold})
1242: within an incremental search toggles the case sensitivity of that
1243: search.  The effect does not extend beyond the current incremental
1244: search, but it does override the effect of adding or removing an
1245: upper-case letter in the current search.
1246: 
1247:   Several related variables control case-sensitivity of searching and
1248: matching for specific commands or activities.  For instance,
1249: @code{tags-case-fold-search} controls case sensitivity for
1250: @code{find-tag}.  To find these variables, do @kbd{M-x
1251: apropos-variable @key{RET} case-fold-search @key{RET}}.
1252: 
1253: @cindex character folding in search
1254: @cindex equivalent character sequences
1255:   Case folding disregards case distinctions among characters, making
1256: upper-case characters match lower-case variants, and vice versa.  A
1257: generalization of case folding is @dfn{character folding}, which
1258: disregards wider classes of distinctions among similar characters.
1259: For instance, under character folding the letter @code{a} matches all
1260: of its accented cousins like @code{@"a} and @code{@'a}, i.e., the
1261: match disregards the diacritics that distinguish these
1262: variants.  In addition, @code{a} matches other characters that
1263: resemble it, or have it as part of their graphical representation,
1264: such as @sc{u+249c parenthesized latin small letter a} and @sc{u+2100
1265: account of} (which looks like a small @code{a} over @code{c}).
1266: Similarly, the @acronym{ASCII} double-quote character @code{"} matches
1267: all the other variants of double quotes defined by the Unicode
1268: standard.  Finally, character folding can make a sequence of one or
1269: more characters match another sequence of a different length: for
1270: example, the sequence of two characters @code{ff} matches @sc{u+fb00
1271: latin small ligature ff}.  Character sequences that are not identical,
1272: but match under character folding are known as @dfn{equivalent
1273: character sequences}.
1274: 
1275: @kindex M-s ' @r{(Incremental Search)}
1276: @findex isearch-toggle-char-fold
1277:   Generally, search commands in Emacs do not by default perform
1278: character folding in order to match equivalent character sequences.
1279: You can enable this behavior by customizing the variable
1280: @code{search-default-mode} to @code{char-fold-to-regexp}.
1281: @xref{Search Customizations}.  Within an incremental search, typing
1282: @kbd{M-s '} (@code{isearch-toggle-char-fold}) toggles character
1283: folding, but only for that search.  (Replace commands have a different
1284: default, controlled by a separate option; see @ref{Replacement and Lax
1285: Matches}.)
1286: 
1287:   Like with case folding, typing an explicit variant of a character,
1288: such as @code{@"a}, as part of the search string disables character
1289: folding for that search.  If you delete such a character from the
1290: search string, this effect ceases.

1299:   Emacs provides several commands for performing search-and-replace
1300: operations.  In addition to the simple @kbd{M-x replace-string}
1301: command, there is @kbd{M-%} (@code{query-replace}), which presents

1307: instead (@pxref{Mark}).  The basic replace commands replace one
1308: @dfn{search string} (or regexp) with one @dfn{replacement string}.  It
1309: is possible to perform several replacements in parallel, using the
1310: command @code{expand-region-abbrevs} (@pxref{Expanding Abbrevs}).

1320: @node Unconditional Replace
1321: @subsection Unconditional Replacement
1322: @findex replace-string
1323: 
1324: @table @kbd
1325: @item M-x replace-string @key{RET} @var{string} @key{RET} @var{newstring} @key{RET}
1326: Replace every occurrence of @var{string} with @var{newstring}.
1327: @end table
1328: 
1329:   To replace every instance of @samp{foo} after point with @samp{bar},
1330: use the command @kbd{M-x replace-string} with the two arguments
1331: @samp{foo} and @samp{bar}.  Replacement happens only in the text after
1332: point, so if you want to cover the whole buffer you must go to the
1333: beginning first.  All occurrences up to the end of the buffer are
1334: replaced; to limit replacement to part of the buffer, activate the
1335: region around that part.  When the region is active, replacement is
1336: limited to the region (@pxref{Mark}).
1337: 
1338:   When @code{replace-string} exits, it leaves point at the last
1339: occurrence replaced.  It adds the prior position of point (where the
1340: @code{replace-string} command was issued) to the mark ring, without
1341: activating the mark; use @kbd{C-u C-@key{SPC}} to move back there.
1342: @xref{Mark Ring}.

1344:   A prefix argument restricts replacement to matches that are
1345: surrounded by word boundaries.

1347:   @xref{Replacement and Lax Matches}, for details about
1348: case-sensitivity in replace commands.

1350: @node Regexp Replace
1351: @subsection Regexp Replacement
1352: @findex replace-regexp
1353: 
1354:   The @kbd{M-x replace-string} command replaces exact matches for a
1355: single string.  The similar command @kbd{M-x replace-regexp} replaces
1356: any match for a specified regular expression pattern (@pxref{Regexps}).
1357: 
1358: @table @kbd
1359: @item M-x replace-regexp @key{RET} @var{regexp} @key{RET} @var{newstring} @key{RET}
1360: Replace every match for @var{regexp} with @var{newstring}.
1361: @end table
1362: 
1363: @cindex back reference, in regexp replacement
1364:   In @code{replace-regexp}, the @var{newstring} need not be constant:
1365: it can refer to all or part of what is matched by the @var{regexp}.
1366: @samp{\&} in @var{newstring} stands for the entire match being
1367: replaced.  @samp{\@var{d}} in @var{newstring}, where @var{d} is a
1368: digit starting from 1, stands for whatever matched the @var{d}th
1369: parenthesized grouping in @var{regexp}.  (This is called a ``back
1370: reference''.)  @samp{\#} refers to the count of replacements already
1371: made in this command, as a decimal number.  In the first replacement,
1372: @samp{\#} stands for @samp{0}; in the second, for @samp{1}; and so on.
1373: For example,
1374: 
1375: @example
1376: M-x replace-regexp @key{RET} c[ad]+r @key{RET} \&-safe @key{RET}
1377: @end example
1378: 
1379: @noindent
1380: replaces (for example) @samp{cadr} with @samp{cadr-safe} and @samp{cddr}
1381: with @samp{cddr-safe}.
1382: 
1383: @example
1384: M-x replace-regexp @key{RET} \(c[ad]+r\)-safe @key{RET} \1 @key{RET}
1385: @end example
1386: 
1387: @noindent
1388: performs the inverse transformation.  To include a @samp{\} in the
1389: text to replace with, you must enter @samp{\\}.
1390: 
1391:   If you want to enter part of the replacement string by hand each
1392: time, use @samp{\?} in the replacement string.  Each replacement will
1393: ask you to edit the replacement string in the minibuffer, putting
1394: point where the @samp{\?} was.
1395: 
1396:   The remainder of this subsection is intended for specialized tasks
1397: and requires knowledge of Lisp.  Most readers can skip it.
1398: 
1399:   You can use Lisp expressions to calculate parts of the
1400: replacement string.  To do this, write @samp{\,} followed by the
1401: expression in the replacement string.  Each replacement calculates the
1402: value of the expression and converts it to text without quoting (if
1403: it's a string, this means using the string's contents), and uses it in
1404: the replacement string in place of the expression itself.  If the
1405: expression is a symbol, one space in the replacement string after the
1406: symbol name goes with the symbol name, so the value replaces them
1407: both.
1408: 
1409:   Inside such an expression, you can use some special sequences.
1410: @samp{\&} and @samp{\@var{n}} refer here, as usual, to the entire
1411: match as a string, and to a submatch as a string.  @var{n} may be
1412: multiple digits, and the value of @samp{\@var{n}} is @code{nil} if
1413: subexpression @var{n} did not match.  You can also use @samp{\#&} and
1414: @samp{\#@var{n}} to refer to those matches as numbers (this is valid
1415: when the match or submatch has the form of a numeral).  @samp{\#} here
1416: too stands for the number of already-completed replacements.
1417: 
1418:   Repeating our example to exchange @samp{x} and @samp{y}, we can thus
1419: do it also this way:
1420: 
1421: @example
1422: M-x replace-regexp @key{RET} \(x\)\|y @key{RET}
1423: \,(if \1 "y" "x") @key{RET}
1424: @end example
1425: 
1426:   For computing replacement strings for @samp{\,}, the @code{format}
1427: function is often useful (@pxref{Formatting Strings,,, elisp, The Emacs
1428: Lisp Reference Manual}).  For example, to add consecutively numbered
1429: strings like @samp{ABC00042} to columns 73 @w{to 80} (unless they are
1430: already occupied), you can use
1431: 
1432: @example
1433: M-x replace-regexp @key{RET} ^.\@{0,72\@}$ @key{RET}
1434: \,(format "%-72sABC%05d" \& \#) @key{RET}
1435: @end example

1446: @vindex replace-lax-whitespace
1447:   Unlike incremental search, the replacement commands do not use lax
1448: space matching (@pxref{Lax Search, lax space matching}) by default.
1449: To enable lax space matching for replacement, change the variable
1450: @code{replace-lax-whitespace} to non-@code{nil}.  (This only affects
1451: how Emacs finds the text to replace, not the replacement text.)
1452: 
1453: @vindex replace-regexp-lax-whitespace
1454:   A companion variable @code{replace-regexp-lax-whitespace} controls
1455: whether @code{query-replace-regexp} uses lax whitespace matching when
1456: searching for patterns.
1457: 
1458: @cindex case folding in replace commands
1459:   If the first argument of a replace command is all lower case, the
1460: command ignores case while searching for occurrences to
1461: replace---provided @code{case-fold-search} is non-@code{nil}.  If
1462: @code{case-fold-search} is set to @code{nil}, case is always significant
1463: in all searches.
1464: 
1465: @vindex case-replace
1466:   In addition, when the @var{newstring} argument is all or partly lower
1467: case, replacement commands try to preserve the case pattern of each
1468: occurrence.  Thus, the command
1469: 
1470: @example
1471: M-x replace-string @key{RET} foo @key{RET} bar @key{RET}
1472: @end example
1473: 
1474: @noindent
1475: replaces a lower case @samp{foo} with a lower case @samp{bar}, an
1476: all-caps @samp{FOO} with @samp{BAR}, and a capitalized @samp{Foo} with
1477: @samp{Bar}.  (These three alternatives---lower case, all caps, and
1478: capitalized, are the only ones that @code{replace-string} can
1479: distinguish.)
1480: 
1481:   If upper-case letters are used in the replacement string, they remain
1482: upper case every time that text is inserted.  If upper-case letters are
1483: used in the first argument, the second argument is always substituted
1484: exactly as given, with no case conversion.  Likewise, if either
1485: @code{case-replace} or @code{case-fold-search} is set to @code{nil},
1486: replacement is done without case conversion.
1487: 
1488: @cindex character folding in replace commands
1489:   The replacement commands by default do not use character folding
1490: (@pxref{Lax Search, character folding}) when looking for the text to
1491: replace.  To enable character folding for matching in
1492: @code{query-replace} and @code{replace-string}, set the variable
1493: @code{replace-char-fold} to a non-@code{nil} value.  (This
1494: setting does not affect the replacement text, only how Emacs finds the
1495: text to replace.  It also doesn't affect @code{replace-regexp}.)

1828: @cindex default search mode
1829: @cindex search mode, default
1830:   The default search mode for the incremental search is specified by
1831: the variable @code{search-default-mode}.  It can be @code{nil},
1832: @code{t}, or a function.  If it is @code{nil}, the default mode is to

1834: and lax-whitespace matches as determined by @code{case-fold-search}
1835: and @code{search-whitespace-regexp}, respectively (@pxref{Lax
1836: Search}).  If the value is @code{t}, incremental search defaults to
1837: regexp searches.  The default value specifies a function that only
1838: performs case folding and lax-whitespace matching.

1841:   The current match of an on-going incremental search is highlighted
1842: using the @code{isearch} face.  This highlighting can be disabled by
1843: setting the variable @code{search-highlight} to @code{nil}.
1844: 
1845: @cindex lazy highlighting customizations
1846: @vindex isearch-lazy-highlight
1847: @cindex lazy-highlight face
1848:   The other matches for the search string that are visible on display
1849: are highlighted using the @code{lazy-highlight} face.  Setting the
1850: variable @code{isearch-lazy-highlight} to @code{nil} disables this
1851: highlighting.  Here are some other variables that customize the lazy
1852: highlighting:
1853: 
1854: @table @code
1855: @item lazy-highlight-initial-delay
1856: Time in seconds to wait before highlighting visible matches.
1857: 
1858: @item lazy-highlight-interval
1859: Time in seconds between highlighting successive matches.
1860: 
1861: @item lazy-highlight-max-at-a-time
1862: The maximum number of matches to highlight before checking for input.
1863: A large number can take some time to highlight, so if you want to
1864: continue searching and type @kbd{C-s} or @kbd{C-r} during that time,
1865: Emacs will not respond until it finishes highlighting all those
1866: matches.  Thus, smaller values make Emacs more responsive.

1869: @vindex search-nonincremental-instead
1870:   Normally, entering @key{RET} within incremental search when the
1871: search string is empty launches a nonincremental search.  (Actually,
1872: it lets you edit the search string, and the next @key{RET} does the
1873: search.)  However, if you customize the variable
1874: @code{search-nonincremental-instead} to @code{nil}, typing @key{RET}
1875: will always exit the incremental search, even if the search string is
1876: empty.
1877: 
1878: @vindex isearch-hide-immediately
1879:   By default, incremental search and query-replace commands match
1880: invisible text, but hide any such matches as soon as the current match
1881: moves off the invisible text.  If you customize the variable
1882: @code{isearch-hide-immediately} to @code{nil}, any invisible text
1883: where matches were found stays on display until the search or the
1884: replace command exits.

1886: @cindex search display on slow terminals
1887: @vindex search-slow-speed
1888: @vindex search-slow-window-lines
1889:   Searching incrementally on slow terminals, such as displays
1890: connected to remote machines over slow connection, could be annoying
1891: due to the need to redraw large portions of the display as the search
1892: proceeds.  Emacs provides a special display mode for slow terminals,
1893: whereby search pops up a separate small window and displays the text
1894: surrounding the match in that window.  Small windows display faster,
1895: so the annoying effect of slow speed is alleviated.  The variable
1896: @code{search-slow-speed} determines the baud rate threshold below
1897: which Emacs will use this display mode.  The variable
1898: @code{search-slow-window-lines} controls the number of lines in the
1899: window Emacs pops up for displaying the search results; the default is
1900: 1 line.  Normally, this window will pop up at the bottom of the window
1901: that displays the buffer where you start searching, bit if the value
1902: of @code{search-slow-window-lines} is negative, that means to put the
1903: window at the top and give it the number of lines that is the absolute
1904: value of that value.
1905:
